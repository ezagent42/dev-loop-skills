# 对比改动与已有覆盖找缺口

> 本文档指导如何读取 coverage-matrix，对比代码改动识别未覆盖的测试路径。

## coverage-matrix 的结构

coverage-matrix artifact（存储在 `.artifacts/coverage/`）记录项目中每个用户场景的测试覆盖状态。典型结构：

```markdown
| 场景 | Unit | E2E | Pre-release | 状态 |
|------|------|-----|-------------|------|
| 创建 agent | Yes | Yes | Yes | 完全覆盖 |
| 发送消息 | Yes | No | Yes | 缺 E2E |
| 重启 agent | No | No | No | 无覆盖 |
```

关键字段：
- **场景**：用户可感知的操作或功能点
- **覆盖层级**：Unit / E2E / Pre-release 各自是否有覆盖
- **状态**：完全覆盖 / 部分覆盖 / 无覆盖

## 对比步骤

### 1. 读取已有覆盖清单

从 coverage-matrix 中提取所有场景及其覆盖状态。建立一个查找表：

```
场景 → {unit: bool, e2e: bool, pre_release: bool}
```

### 2. 映射改动到场景

将 diff-analysis（Step 2 产出的影响清单）中的"影响流程"映射到 coverage-matrix 中的场景。

映射规则：
- **精确匹配**：影响流程名称直接对应场景名称
- **模糊匹配**：影响流程的关键词出现在场景描述中
- **新增场景**：影响流程在 coverage-matrix 中没有对应条目——这是新功能，需要从零建立覆盖

### 3. 标注缺口类型

对每个受影响的场景，标注缺口类型：

| 缺口类型 | 条件 | 测试需求 |
|----------|------|----------|
| **完全缺口** | 场景无任何层级覆盖 | 需要新增 E2E 用例 |
| **E2E 缺口** | 有 Unit 但无 E2E | 需要补充 E2E 用例 |
| **新增场景** | 场景不在 matrix 中 | 需要新增 E2E 用例 + 更新 matrix |
| **覆盖充分** | 场景已有 E2E 覆盖 | 仅需回归验证，不需要新用例 |

### 4. 标注回归风险

已有 E2E 覆盖的场景，如果被代码改动影响，存在回归风险：

- **直接回归**：改动直接修改了该场景的实现代码
- **间接回归**：改动修改了该场景依赖的公共模块（如协议、配置）
- **配置回归**：改动修改了测试环境配置

回归风险场景不需要新增用例，但需要确保已有用例在新代码上通过。在测试计划中标注为"回归验证"，优先级通常为 P0。

### 5. 产出缺口清单

```
## 覆盖缺口清单

### 完全缺口（需新增 E2E）
1. 场景：重启 agent → 无任何覆盖，改动影响了重启逻辑
2. 场景：agent 命名冲突处理 → 新增场景，matrix 中无对应

### E2E 缺口（需补充 E2E）
1. 场景：发送消息 → 有 Unit 无 E2E，改动影响了消息路由

### 回归风险（需验证已有用例）
1. 场景：创建 agent → 有 E2E 覆盖，但改动修改了 agent 创建流程

### 覆盖充分（无需额外操作）
1. 场景：列出 agent → 有完整覆盖，改动未影响此流程
```

## 没有 coverage-matrix 时的处理

如果项目还没有 coverage-matrix（Skill 0 未运行或产出未注册），无法做精确对比。此时的策略：

1. 将所有受影响场景视为"覆盖状态未知"
2. 为每个场景都生成测试用例，优先级根据改动风险确定
3. 在测试计划的风险标注中明确说明"缺少 coverage-matrix，无法判断覆盖缺口"
4. 建议在测试计划确认后，补充 coverage-matrix（触发 Skill 0）

## 注意事项

- coverage-matrix 可能过时——如果最后更新时间远早于代码改动，需要在风险标注中提醒
- 不要只看 E2E 缺口——有时 Unit 覆盖已经足够，不需要为每个场景都写 E2E
- 新增场景容易被遗漏——如果 code-diff 中有新文件或新导出函数，很可能是新场景
- 删除场景同样重要——如果功能被移除，对应的测试用例应该标注为"待清理"
