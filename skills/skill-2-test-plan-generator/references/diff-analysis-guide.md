# 从 code-diff 提取影响范围

> 本文档指导如何分析 code-diff artifact，确定受影响的模块和用户流程。

## code-diff 的结构

code-diff artifact（存储在 `.artifacts/code-diffs/`）通常包含：
- 变更文件列表（路径 + 改动类型）
- 改动摘要（功能描述）
- 关联的 commit 或 PR 信息

## 分析步骤

### 1. 提取变更文件列表

从 code-diff 的正文中提取所有变更的文件路径。每个文件标注改动类型：

| 类型 | 含义 | 测试影响 |
|------|------|----------|
| A (Added) | 新增文件 | 需要新增功能测试 |
| M (Modified) | 修改文件 | 需要验证修改行为 + 回归测试 |
| D (Deleted) | 删除文件 | 需要验证依赖方不受影响 |
| R (Renamed) | 重命名 | 需要验证引用更新 |

### 2. 从文件路径定位模块

文件路径直接反映所属模块。以 zchat 项目为例：

| 路径前缀 | 模块 | 说明 |
|----------|------|------|
| `zchat/cli/` | CLI | 命令行接口 |
| `zchat-channel-server/` | Channel Server | MCP ↔ IRC 桥接 |
| `zchat-protocol/` | Protocol | 协议定义 |
| `weechat-zchat-plugin/` | WeeChat Plugin | WeeChat 插件 |
| `tests/unit/` | Unit Tests | 单元测试 |
| `tests/e2e/` | E2E Tests | 端到端测试 |
| `*.toml` / `*.yaml` | Config | 配置文件 |

对于其他项目，根据目录结构推断模块划分。常见的模式：
- `src/` 下的子目录通常对应不同模块
- `lib/` 下按功能分组
- 顶层目录名通常是独立组件

### 3. 识别受影响的用户流程

不只看改了哪个文件，还要想这个改动影响了用户的哪些操作。方法：

**向上追溯**：从改动的函数/类出发，找到调用方，直到用户可感知的入口。例如：
- 改了 `irc_client.py` 的连接逻辑 → 影响"启动 agent"、"重启 agent"流程
- 改了 `naming.py` 的 `scoped_name()` → 影响所有涉及 agent nick 的操作

**向下追溯**：从改动的函数出发，找到依赖方。例如：
- 改了协议格式 → 所有解析该协议的组件都受影响

### 4. 评估改动风险等级

按影响范围和严重程度分级：

| 风险等级 | 条件 | 测试优先级 |
|----------|------|-----------|
| 高 | 核心路径（消息收发、agent 生命周期）、跨模块影响 | P0 |
| 中 | 单模块内部改动、配置变更 | P1 |
| 低 | 文档、注释、格式化、仅测试代码改动 | P2 |

### 5. 产出格式

分析完成后，产出一份按模块组织的影响清单：

```
## 影响清单

### 模块：CLI
- 改动文件：zchat/cli/agent.py (M)
- 改动类型：修改 agent create 逻辑
- 影响流程：创建 agent、重启 agent
- 风险等级：高

### 模块：Protocol
- 改动文件：zchat-protocol/naming.py (M)
- 改动类型：修改命名规则
- 影响流程：所有涉及 agent nick 的操作
- 风险等级：高
```

## 注意事项

- 不要仅看文件名就判断影响范围——需要理解改动的语义
- 配置文件改动容易被低估——一个 config 变更可能影响所有模块
- 测试文件本身的改动也要分析——可能说明了哪些行为发生了变化
- 删除文件时要特别关注——可能有未更新的引用
